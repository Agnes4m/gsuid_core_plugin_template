# GSUID Plugin Template 架构方案

## 概述

本文档详细说明 GSUID Plugin Template 的双轨制架构方案，支持两种使用方式：
1. **GitHub Template** - 独立仓库方式
2. **框架内置** - 本地复制方式

## 方案对比

### GitHub Template 方式

**适用场景：**
- 新手开发者第一次创建插件
- 计划开源发布的插件
- 需要独立版本控制的插件
- 团队协作开发

**优势：**
- ✅ 可视化操作，零命令行基础
- ✅ 完整的 Git 历史记录
- ✅ 独立的 Issue/PR 管理
- ✅ 可直接配置 GitHub Actions
- ✅ 便于社区发现和贡献

**劣势：**
- ❌ 需要手动同步上游模板更新
- ❌ 额外的仓库管理成本

**使用流程：**
```
1. 访问模板仓库
2. 点击 "Use this template" 按钮
3. 填写仓库信息，创建新仓库
4. Clone 到本地 plugins 目录
5. 运行 python init_plugin.py
6. 开始开发
```

### 框架内置方式

**适用场景：**
- 框架核心开发者测试
- 快速原型验证
- 企业内部开发
- 不需要独立发布的插件

**优势：**
- ✅ 模板更新自动同步
- ✅ 无需管理额外仓库
- ✅ 继承框架的 CI/CD 配置
- ✅ 快速开始，一步到位

**劣势：**
- ❌ 不方便独立发布
- ❌ 版本历史与框架混杂
- ❌ 不适合社区协作

**使用流程：**
```
1. Clone 框架仓库
2. 复制模板目录
3. 重命名并运行初始化脚本
4. 开始开发
5. 可选择性提交 PR 到框架
```

## 决策矩阵

| 因素 | GitHub Template | 框架内置 |
|------|----------------|----------|
| 上手难度 | ⭐ 低 | ⭐⭐ 中 |
| 版本控制 | 独立完整 | 依赖框架 |
| 模板更新 | 手动同步 | 自动跟随 |
| 社区贡献 | 独立管理 | 集中到框架 |
| CI/CD | 需单独配置 | 继承框架 |
| 发布插件 | 直接发布 | 需分离代码 |
| 学习曲线 | 平缓 | 较陡 |

## 推荐选择

| 用户类型 | 推荐方式 | 理由 |
|---------|---------|------|
| 第一次开发插件 | GitHub Template | 简单直观，有完整示例 |
| 要发布开源插件 | GitHub Template | 独立仓库便于社区贡献 |
| 框架核心开发者 | 框架内置 | 需要同步更新模板 |
| 企业内部开发 | 框架内置 | 可定制内部模板 |
| 快速验证想法 | 框架内置 | 无需创建新仓库 |
| 学习模板结构 | 两种方式均可 | 根据偏好选择 |

## 技术细节

### GitHub Template 自动化

当用户使用模板创建仓库时，自动执行：
1. 删除模板专属文件（如 template-cleanup.yml）
2. 可选：自动运行 init_plugin.py
3. 创建初始提交

### 版本同步策略

**GitHub Template → 框架内置：**
- 通过 PR 将模板更新同步到框架
- 定期（如每月）检查并同步

**框架内置 → GitHub Template：**
- 框架核心改进通过 PR 提交到模板仓库
- 紧急修复优先同步

## 结论

双轨制方案兼顾了不同用户群体的需求：
- **GitHub Template** 面向广大插件开发者，提供最佳的上手体验
- **框架内置** 面向核心开发者，便于模板迭代和测试

两种方案共享相同的模板核心内容，确保一致性。
